## How to continue

**Saturation warning as of batch 6 (2026-07-20):** at 266 recipes, yield has dropped sharply — batch 6 wrote only 11 against a 50-recipe target (vs. 29/42/56/50/62 in batches 5/4/3/2/1). A second scroll session in the same sitting, pushed all the way to the magazine's own end-of-feed footer (~144 more URLs), found only 2 surviving candidates after filtering — both already-known skips. The magazine's practically-reachable range per session (roughly 230-400 unique stories; Flipboard's infinite scroll restarts from the top every session and eventually hits a real footer, it's not truly infinite per-session) is now substantially covered by already-converted recipes. **Recommendation for batch 7 and beyond:** don't expect 50-recipe yields from a single scroll session anymore. Either (a) run multiple scroll sessions and accept a smaller net yield per sitting, tracked cumulatively over several conversations, or (b) ask the user whether they want to add a second source magazine/feed alongside this one, since this specific magazine's low-hanging fruit is largely picked over.

Say "convert the next Flipboard batch" (or similar, e.g. "convert 40 more recipes"). Whoever picks this up should:
1. Open https://flipboard.com/@enetoe/food-flavors-acbn0akgy in Claude in Chrome and scroll. **Important finding from batch 3:** this feed does not appear to support resuming from a specific scroll position/index across sessions — a fresh scroll re-surfaces much of the same range already covered by earlier batches (confirmed via exact `source_url` matches). Budget for scrolling substantially further this time (e.g. past 150–200 mouse-wheel scroll actions) before expecting mostly-new territory, and extract early anyway — genuinely new candidates keep turning up even within an overlapping range, since earlier batches only dispatched candidates that passed the single-recipe/non-duplicate filter, leaving other cards from the same range unprocessed.
2. **Efficient extraction method (established in batch 3):** don't read the accessibility tree card-by-card. Instead, run a small JS snippet via `javascript_tool` that selects `a[href*="/redirect"]`, decodes the `?url=` query param via `new URL(href).searchParams.get('url')`, and walks up each anchor's ancestors to find the nearest `h1`–`h4` for the title. Accumulate results in a page-global object (e.g. `window.__collected`) across scroll bursts so nothing is lost as the DOM virtualizes distant cards. To read back more than ~15-20 collected entries at once (the JS tool's own output truncates well before that), don't return the array directly — instead write the joined text into a `<pre>` element, replace `document.body.innerHTML` with just `<article><pre>...</pre></article>` (article tag matters — `get_page_text` prioritizes it), then call `get_page_text`, which has a much larger limit and returned all 139 collected entries in one call.
3. Use real mouse-wheel `scroll` actions (`computer` tool, `scroll_amount: 15`, in bursts of 2-5 followed by `wait(2-3)`) — programmatic `window.scrollTo()` does not trigger Flipboard's infinite-scroll loading. Expect occasional "Page still loading" errors from `browser_batch` mid-scroll (the infinite-scroll XHR briefly blocks navigation-state checks) — just wait longer and retry the same scroll, no data is lost.
4. **Run the full duplicate-detection workflow in `_cookbook-meta-prompt.md` §6c on every collected URL before converting anything** — exact `source_url` match against all existing recipes, same-domain+near-identical-title fuzzy match (gamintraveler.com and 30seconds.com both republish articles under new date-paths/tip-IDs — for 30seconds.com specifically, match on the numeric tip ID in the URL, since the slug text can change between republishes even when the ID doesn't), and a check for duplicates within the newly-collected batch itself. **New in batch 3:** near-identical dish names from a *different* source domain are not automatically duplicates — e.g. two distinct Greek-lemon-potato recipes, two distinct lemon pastas, and two distinct tzatziki recipes were all kept as legitimate separate entries after actually comparing ingredients/technique. Always compare content, not just the dish name.
5. **Known persistent blocks via raw fetch (as of batch 4, 2026-07-19):** eatingwell.com, simplyrecipes.com, and receitas.globo.com are still blocked by `WebFetch`/`web_fetch` specifically (`URL is on blocklist`), and gamintraveler.com sometimes hits a Cloudflare bot-check wall the same way. **Important finding from batch 4: none of these are actually unreachable — they were all recovered by using the Claude-in-Chrome browser tools (`navigate` + `wait` + `get_page_text`) instead of raw fetch.** Two recovery passes in batch 4 wrote 18 recipes this way, including several eatingwell.com/simplyrecipes.com/receitas.globo.com pages and gamintraveler.com URLs that had failed raw fetch (some of the same specific 2024-dated gamintraveler URLs flagged as failed in batch 3 — Tuscan chicken 2024/12/30 and chicken tikka masala 2024/12/22 — succeeded via the browser in batch 4, so that batch-3 "possibly transient" note is resolved: it's a fetch-method issue, not a content-availability one). **Recommendation for batch 5 and beyond: default straight to the browser for these four domains rather than trying raw fetch first** — it will save a recovery pass. bonappetit.com remains a genuine sandbox-level block (403) even via browser, not yet retried since batch 3.
6. Split the batch across parallel sub-agents (~12-15 URLs each has worked well across all three batches) with the same schema/anti-fabrication/duplicate-detection instructions — see `_cookbook-meta-prompt.md` §5–§6d, and quote any `source_title` that contains a colon (YAML gotcha hit multiple times now).
7. After all sub-agents finish, rebuild (`python3 _build/build.py` — if this throws a `SyntaxError` pointing at a line that looks fine when read directly, that's the OneDrive/bash staleness bug described in §13, not a real syntax error; use the scratch-copy workaround — batch 3's build completed cleanly on the first try with no staleness issue), update `README.md`'s recipe index (a Python script that parses every recipe's frontmatter and regenerates the whole "Current recipes" section in one pass, rather than hand-editing, worked well in batch 3 — see the approach in `_build/build.py` for the frontmatter-parsing pattern), and write `flipboard-batch-N-report.md` alongside this file.
